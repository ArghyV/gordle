# gordle

> **⚠️ No human review. Not actively maintained.**
> This project was generated by an LLM pipeline with no manual code inspection. It may contain bugs, security issues, or incomplete behaviour. Use at your own risk.

---

gordle is a terminal tool that drives an AI-powered Go code generation pipeline. You give it a plan file; it calls an LLM to break the plan into tasks, generates Go code for each task, validates the code inside a throwaway Docker container, and saves passing outputs to a versioned store.

It is a **meta-tool**: it writes Go code, it does not implement application logic itself.

---

## Prerequisites

- Go 
- Docker (running locally — required for the validator)
- A [Cerebras](https://cerebras.ai) API key

---

## Setup

```bash
git clone https://github.com/ArghyV/gordle.git
cd gordle
echo "CEREBRAS_API_KEY=your_key_here" > .env
```

`.env` will be added to `.gitignore` automatically on first run.

---

## Usage

```bash
go run ./main --plan plan.md              # default: confirms each task before running
go run ./main --plan plan.md --yolo       # skip confirmations
go run ./main --plan plan.md --max-retries 3
```

| Flag | Default | Meaning |
|---|---|---|
| `--plan` | *(required)* | Path to the plan file |
| `--yolo` | `false` | Skip per-task confirmation prompts |
| `--max-retries` | `2` | How many times to retry a failing task |

## What the code does vs. what the prompts do

This project has two distinct layers.

### Code — the harness (`main`, `queue`, `llm`, `artifacts`, `injector`, `validator`)

The Go code is the **plumbing**. It does not make any decisions about what to generate.

| Package | Responsibility |
|---|---|
| `types` | Shared data structures and interface contracts. No logic. |
| `main` | CLI flags, spinner, per-task confirmation prompt, top-level orchestration. |
| `queue` | Decomposes a plan file into `[]TaskSpec` via LLM call; topological sort of dependencies; dispatch loop with retry. |
| `llm` | HTTP client for the Cerebras API; SHA-256 keyed response cache; `.env` loader. |
| `artifacts` | Versioned file store under `.harness/artifacts/`. Reads and writes generated code by ref + version number. |
| `injector` | Builds the prompt string sent to the code-generation LLM: task spec + artifact contents + grep index + prior error output. |
| `validator` | Runs the generated code's validation command (`go build`, `go vet`, `go test`) inside an ephemeral Docker container, then destroys it. |

### Prompts — the intelligence (`decomposition-prompt.md`, `worker-system-prompt.md`, `plan.md`)

The prompts are the **decision layer**. They are plain text files, not compiled code.

| File | Role |
|---|---|
| `plan-header.md` | A formal list of all required info gordle needs to process your plan. Prepend this to any freeform plan to create a plan file. |
| `plan.md` | The specification you write. Describes what should be built, which packages are involved, what interfaces must be satisfied, and what order tasks should run in. |
| `decomposition-prompt.md` | System prompt for the *decomposition* LLM call. Instructs the model to validate the plan header and break the plan body into an ordered list of `TaskSpec` YAML blocks. |
| `worker-system-prompt.md` | System prompt for the *code generation* LLM call (one per task). Instructs the model to produce a single, complete, immediately writable Go source file — no prose, no markdown fences. |

The harness sends `decomposition-prompt.md + plan.md` to the LLM to produce task specs, then sends `worker-system-prompt.md + assembled context` for each task to produce source files.

## Output layout

```
.harness/
  artifacts/
    code_output/    generated .go files, versioned (e.g. store_v1.go)
    task_specs/     decomposed task spec YAML files
    design_doc/     snapshots of the plan file
  cache/            {sha256_of_prompt}.json  — cached LLM responses
```

---

## Security notes

- Generated code runs **only** inside an ephemeral Docker container with no network access.
- The container image is `golang:1.22-alpine`. It is destroyed after every run.
- Your API key is never logged. It is read from `.env` and held in memory only.
- The LLM API call itself is plain HTTPS — it is **not** containerised.

---

## Known limitations

- No parallel task execution (serial only).
- No fallback LLM provider (Cerebras only; fallback disabled by design during prototyping).
- Integration tests use in-memory stubs; they do not make real LLM or Docker calls.
- No human has reviewed the generated code in this repository.
